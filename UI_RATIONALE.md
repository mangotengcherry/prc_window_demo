# Process Window 대시보드 — UI 요소별 도입 배경·설계 근거 (교차검증용)

> **이 문서의 목적**
> 대시보드의 각 UI 요소가 **왜** 도입됐고, 어떤 **데이터·통계 근거**에 기반하며, 어떤 **가정·한계**를 갖는지 명시한다.
> GPT 등 다른 모델이 코드 없이도 논리적 타당성을 **독립 검증**할 수 있도록, 항목마다 `교차검증 포인트`(반증/확인 질문)를 둔다.
> 표기: `μ`=평균, `σ`=표준편차, `Φ`=표준정규 누적분포. spec=규격(DC/user), 관리한계=데이터 기반 ±3σ.

---

## 0. 데이터 모델 전제 (모든 설계의 근간)

이 전제를 이해하지 못하면 아래 절반은 "왜 이렇게 했나"가 풀리지 않는다.

- **Long fact table**: 1행 = `(wafer, fab_step)`. 한 wafer가 여러 fab_step을 거치므로 wafer당 여러 행.
- **X feature (FAB/계측)**: 공정·계측에서 나오는 설명변수. 확보 시점 = `fab_track_out_time`(공정 trackout). 거의 즉시 확보.
- **Y target (EDS)**: 최종 전기검사(EDS) 결과(예: BIN 수율). 확보 시점 = `eds_tkout_time`.
- **★ 핵심 비대칭 — 시간 lag ★**: Y는 fab trackout 후 **약 50~60일(wafer마다 변동)** 뒤에야 확보된다.
  → **최근 ~50–60일 구간은 X(feature)는 있는데 Y(target)가 아직 없다(미확보).**
  이 비대칭이 "추정 y", "관리이탈 예상", "미확보 gap 음영"이 존재하는 **유일한 이유**다.
- **`observed` 플래그**: 그 행의 Y가 확보됐는지. `observed=False`면 target=NaN(추정 대상).
- **식별자**: `root_lot_id`(=wafer 묶음 lot) ⊃ `wafer_id`. scatter 점 1개 = wafer 1매.
- **용어 구분(중요)**:
  - **관리한계(control limit)** = 데이터로부터 산출한 `μ ± 3σ`. "이 공정이 평소와 다른가"(통계적 안정성).
  - **spec(규격)** = 합격 기준. `DC spec`(장비/공정 기본 규격, 항상 존재) vs `user spec`(사용자가 입력하는 평가 기준).
  - **σ_단기(within)** = 이동범위 기반 `MR̄/1.128` → Cpk용(군내 산포). **σ_전체(overall)** = 표본 표준편차 → Ppk용(장기 산포).

## 1. 좌측 조건 패널 (Sidebar)

### 1.1 FAB_STEP → X feature 종속 (선택 시 X 초기화 + 안내)
- **무엇**: fab_step을 바꾸면 X feature 후보가 그 step에 매칭되는 것만 노출되고, 기존 X 선택은 비워진다(조용히 지우지 않고 알림).
- **도입 배경**: feature는 특정 fab_step에 종속된 계측 항목이다. 다른 step의 feature를 섞으면 무의미한 조합이 만들어진다. "조용한 초기화"는 사용자가 영문 모르고 빈 결과를 보는 혼란을 막기 위함.
- **근거**: `feature_catalog`(=fab_metro_prc)의 `feature_key`가 fab_step에 1:1로 매핑.
- **가정·한계**: matching=on 가정. 사용자가 일부러 비매칭 feature를 보려면 matching을 꺼야 함.
- **교차검증 포인트**: fab_step과 무관한 feature를 X로 허용하면 binning/회귀가 의미를 갖는가? (아니오 — 그래서 종속.)

### 1.2 Category → Y target 종속, 기간 2종 (fab / eds)
- **무엇**: category가 Y target 후보를 결정. 기간 필터가 **두 개** — ① fab 기간(`fab_track_out_time`, 분석 모집단·x축) ② y확보 기간(`eds_tkout_time`).
- **도입 배경**: 0절의 lag 때문에 "언제 공정한 wafer인가"와 "언제 결과가 나온 wafer인가"는 다른 축이다. 둘을 한 기간으로 묶으면 미확보 구간을 표현할 수 없다.
- **가정·한계**: y확보 기간을 좁히면 표본이 급감할 수 있음(최근일수록 미확보 多).
- **교차검증 포인트**: 분석 기간을 fab 기준으로만 잡고 target을 관측-only로 거르면, 최근 구간 target 평균이 **생존 편향**(먼저 확보된 wafer만)으로 왜곡될 수 있는가? → 그래서 추정으로 보완.

### 1.3 분할(category feature) — split vs multi-line
- **무엇**: PPID/EQP/ECO 등 wafer 단위 속성으로 데이터를 나눠 본다. `split`(값별 별도 행) / `multi_line`(한 차트에 겹쳐 비교, spec 공유).
- **도입 배경**: "어느 설비/레시피가 더 나은가"는 분할 비교 없이는 답할 수 없음(→ 4.3 조건 비교와 연결).
- **가정·한계**: 분할이 잘면 셀당 표본 부족. multi_line은 추정 y 미표시(분리 모드에서만).

### 1.4 그룹 target (합산 합성 target)
- **무엇**: 여러 원본 target을 합산한 합성 target을 인라인 정의(요청에 동봉, 클라이언트 보유).
- **도입 배경**: "BIN0131+0132+0133 합"처럼 의미 단위가 원본 컬럼과 다를 때. 서버 상태 없이(stateless) 재현 가능하게.
- **가정·한계**: 현재 `sum`만, **`min_count=len(srcs)`** → **모든 source가 있어야 합산**, 하나라도 결측이면 합=NaN(부분합으로 인한 불완전 합산 방지). 이전 `min_count=1`(있는 값만 합산)에서 변경 — SUM 지표의 정합성 우선. (데모는 target이 wafer 단위 all-or-nothing이라 동작 차이 없음; 실데이터의 부분 결측에서만 효과.)

## 2. 상단 바 (Topbar)

### 2.1 Provenance 바 (출처·기간·표본·쿼리ID)
- **무엇**: 현재 결과가 어떤 조건/기간/표본에서 나왔는지 + 결정적 쿼리 ID.
- **도입 배경**: 분석 결과의 **재현성·추적성**. 캡처 한 장만 봐도 무슨 조건인지 알 수 있어야 의사결정에 쓸 수 있다.
- **교차검증 포인트**: 같은 조건 → 같은 쿼리ID(FNV-1a 해시)인가? 표본 수가 KPI/차트와 일치하는가?

### 2.2 🔗 공유 / ⬇ CSV / 추정 y 토글
- **공유**: 조건을 base64로 URL(`?q=`)에 직렬화 → 같은 링크로 동일 화면 재현.
- **⬇ CSV (요청 기능)**: 현재 조건의 **wafer 원시행**을 내려받음. 컬럼 = 식별자(root_lot_id·wafer_id)+시간+observed+분할+선택 feature/target. 관측·미관측 **모두**, brush 선택 반영, UTF-8 BOM(엑셀 한글).
  - **도입 배경**: 화면 집계만으로는 "이 점이 어느 wafer인지" 원장 추적·외부 재분석이 안 됨. raw 제공으로 검증 루프를 닫음.
  - **가정·한계**: 미확보 행의 target은 빈 값(NaN). 다운샘플과 무관하게 전수 행.
- **추정 y 토글**: 미관측 wafer의 추정 y 오버레이 on/off(분리 모드).

## 3. KPI 밴드

각 KPI는 "지금 이 조건에서 가장 먼저 봐야 할 한 숫자"를 노출한다.

| KPI | 정의 | 도입 배경 / 해석 |
|---|---|---|
| 조합 | 표시된 (feature×target) 조합 수 | 분석 규모 파악 |
| 표본 수(기간) | 기간 내 wafer 행 수(관측+미관측) | 신뢰도 가늠 |
| 최저 Cpk(DC) | 모든 조합의 Cpk(DC spec, 단기σ) 중 **최솟값** | 가장 취약한 feature를 한 숫자로. <1 위험(빨강)·<1.33 주의 |
| 최저 Cpk(user) | user spec 입력 조합의 Cpk 최솟값 | 사용자 기준 능력 |
| 표본 부족 조합 | 최대 bin 표본 < `min_n`인 조합 수 | 평균이 불안정한 조합 경고 |
| **관리이탈 예상** | lag 미확보 구간 추정치가 target 관리한계(±3σ)를 벗어날 것으로 예측되는 wafer 수 + 추정 평균 이동(σ) | 0절의 lag 때문에 결과가 아직 없을 때, **선제 경보**(→ 5.3) |

- **"최저"를 쓰는 이유**: 공정능력은 "가장 약한 고리"가 지배한다(평균은 약점을 가림). worst-case 노출이 의사결정에 안전.
- **교차검증 포인트**: 최저 Cpk가 인사이트 패널(4.2)의 1위(가장 약한 feature)와 일치하는가? "관리이탈 예상"의 σ shift가 5.3 배지와 같은 값인가?

## 4. 인사이트 패널 (조합 위)

### 4.1 영향 요인(driver) 랭킹
- **무엇**: 선택 target별로, 같은 fab_step의 numeric feature를 **|상관계수|** 내림차순으로.
- **도입 배경**: "이 수율을 움직이는 게 무엇인가"를 먼저 보여줘야 어떤 feature의 window를 봐야 할지 정해진다.
- **근거**: 관측 wafer에서 `corr(feature, target)`(Pearson), 양쪽 non-null, `n≥5`. 부호 포함 표기(막대 색=부호).
- **가정·한계**: **상관 ≠ 인과**. 선형 상관만(비선형/교호작용 누락). 공선성 미보정.
- **교차검증 포인트**: 상위 driver가 도메인 상식과 맞는가? n이 작은 상관을 과신하지 않는가? 부호가 window 차트 기울기와 일치하는가?

### 4.2 공정능력(Cpk) 요약
- **무엇**: feature별 `Cpk(DC)`를 **약한 순**으로. (target과 무관 — Cpk는 feature가 자기 spec 대비 안정한지이므로 feature 단위가 정직.)
- **도입 배경**: driver(4.1)로 "무엇이 중요"한지 알았으면, 그 feature가 "지금 관리되고 있나"를 봐야 한다.
- **근거**: `Cpk = min(USL−μ, μ−LSL) / (3·σ_단기)`, σ_단기 = `MR̄/1.128`.
- **가정·한계**: σ_단기는 시간 정렬(이동범위) 가정. 분포 정규성 암묵 가정.
- **교차검증 포인트**: feature×target 매트릭스가 아니라 feature 리스트인 게 맞나? (맞음 — Cpk(DC)는 target 독립이라 매트릭스면 열이 중복.)

### 4.3 조건 비교 랭킹
- **무엇**: 분할(EQP 등) 적용 시 (feature×target)별로 분할값을 **target 평균순**으로 + 각 조건의 Cpk.
- **도입 배경**: "어느 설비/레시피/조건이 best-known-method인가"를 직접 순위로.
- **가정·한계**: 평균만으로 줄세움(분산·표본수는 보조 표기). 교란요인(혼합) 미보정 — 관측 비교지 실험 아님.
- **교차검증 포인트**: 1위 조건이 표본도 충분하고 Cpk도 양호한가, 아니면 표본 적은 우연인가?

## 5. 조합별 카드 (ComboRow) — feature × target 1개

### 5.1 Window 차트 (y vs x, binning)
- **무엇**: X feature를 구간(bin)으로 나눠 각 구간의 Y target 평균(+표본수/신뢰구간)을 막대·선으로. spec/DC 라인 표시.
- **도입 배경**: "X를 어느 범위로 가져가면 Y가 좋은가" = **process window**의 직접 시각화(대시보드의 본질).
- **근거**: `pd.cut`로 등간격 bin, 관측 wafer만, bin별 `mean/count/std`, SEM=std/√n.
- **가정·한계**: 관측-only(미확보 제외) → 최근 구간 과소대표 가능. 등간격 bin(분위 아님).
- **교차검증 포인트**: bin 표본이 충분한가(표본 부족 표식)? window가 spec 안에 있는가?

### 5.2 capability strip (Cpk/Ppk vs DC/user)
- **무엇**: 이 조합 feature의 `Cpk/Ppk`를 DC·user spec 기준으로.
- **근거**: Cpk=단기σ, Ppk=전체σ. `Ppk≤Cpk`면 군간 변동(드리프트) 존재 신호.
- **교차검증 포인트**: Cpk와 Ppk 격차가 크면 5.4 drift와 정합하는가?

### 5.3 시계열 차트 (상=target, 하=feature) — 카드의 핵심
한 차트에 여러 레이어가 겹친다. 각 레이어의 도입 배경:

- **target/feature scatter(점=wafer 1매)** + **추세선(이동평균)**: 시간에 따른 거동. 추세선은 국소 평균(노이즈 제거).
- **관리한계(UCL/LCL, ±3σ)**: 통계적 안정성 기준선. 라벨은 선 우측 끝(좌측 y축 이름과 겹침 방지).
- **DC/user spec 라인**: 규격 기준선(관리한계와 색·dash로 구분).
- **★ 미확보 gap 음영 ★**: 마지막 관측 target 시점 ~ 마지막 feature 시점. 0절 lag로 Y가 아직 없는 구간을 시각적으로 못박음("미확보(추정 대상)"/"추정 구간").
- **추정 y(빈 다이아몬드)**: 미관측 wafer의 X로 회귀 예측한 Y(관측값과 표식 구분).
- **★ 추정 평균 비교(직전 요청) ★**: 관측 구간 평균 vs 추정 구간 평균을 **두 수평 세그먼트**로. 팀 컨벤션상 추정이 높으면 빨강. 라벨에 값+차이(Δ).
  - **도입 배경**: "추정 구간이 평소(관측)와 얼마나 다른가"를 한눈에. 숫자(σ shift)만으론 체감이 안 됨.
- **drift 배지(⚠ 추세 ↑/↓ ±Xσ)**: feature 최근 평균이 기준 대비 이동.
- **관리이탈 예상 배지**: 이 조합의 lag 기반 예보(미확보 N장 중 관리한계 초과 예측 수 + σ shift). **신뢰 가드**: 추정 `R²<0.2` 또는 **외삽≥50%**면 빨강 경보 대신 **회색 "≈ … 참고용"**으로 강등(외삽 비율은 추정 R² 배지·툴팁에 상시 표기). → 저신뢰 예보를 단정 경보로 오인하지 않게.
- **wafer 식별 툴팁(요청)**: 점 hover 시 `root_lot_id·wafer_id·tkout`(관측 target은 EDS확보 시각도, 추정은 "(추정)" 표식).

**근거(공식)**:
- 추정: 관측 wafer로 `OLS: y = a·x + b` 적합(`n≥5`), 미관측 wafer의 x로 ŷ 예측. R² 표기.
- drift: `n≥20`, `k=max(5, ⌊n/5⌋)`, `shift = (mean(최근 k) − mean(앞 n−k)) / σ`, `|shift|≥1`이면 flag.
- 관리이탈 예보: 관측 target에서 `UCL/LCL=μ±3σ` 산출 → ŷ가 `[LCL,UCL]` 밖이면 카운트. `σ_shift=(mean(ŷ) − mean(y관측))/σ관측`.

**가정·한계(중요)**:
- 추정은 **동시점 X–Y 회귀**(관측 wafer 기반). 최근 feature가 drift로 관측 X 범위를 벗어나면 **외삽**(신뢰↓, R² 낮으면 ⚠).
- 관리한계를 **관측 target**에서 잡고 추정값을 그 한계로 판정 → 별도 target spec이 없을 때의 차선책. target 규격이 따로 있으면 그 기준이 더 엄밀.
- 추정 평균/예보는 lag 가정(50~60일)과 미확보 모집단에 의존.
- **교차검증 포인트**:
  - 추정 R²가 낮은데 관리이탈 예보를 신뢰해도 되나? (R²·외삽 여부 함께 봐야.)
  - drift의 80/20 분할이 과적합/둔감 사이에서 적절한가? 다른 분할이면 결론이 바뀌나?
  - "관측 평균 vs 추정 평균" 차이가 feature drift(5.4)와 인과적으로 연결되는가, 우연인가?

### 5.4 user spec 입력
- **무엇**: lower/upper 입력 시 그 기준의 Cpk/Ppk/in-spec% 산출(디바운스).
- **근거**: in-spec% = `Φ((U−μ)/σ) − Φ((L−μ)/σ)` (**정규근사**).
- **가정·한계**: 정규성 가정 — 치우친 분포면 과/소추정. 그래서 "근사" 라벨.

## 6. 요약 테이블 (DataTable)
- **무엇**: 조합별 μ·σ(전체/단기)·n·spec·Cpk 등 수치 원장.
- **도입 배경**: 차트로 본 패턴을 **숫자로 확인**(차트는 인지, 표는 검증).
- **교차검증 포인트**: 표의 μ/σ가 차트의 추세·관리한계와 산술적으로 정합하는가?

## 7. 교호작용 분석 (Interaction)

window(5.1)와 **독립**. X feature·Y target 중 2개를 골라 x/y축으로, 세 번째를 값(color)으로.

### 7.1 Scatter (value=color, 영역 focus)
- **무엇**: 두 변수의 산점 + 값을 색(팀 컨벤션: 높음=빨강, 낮음=회색). 영역 드래그(brush)=그 구간 데이터로 **재계산**(focus).
- **도입 배경**: 두 인자의 **교호작용**(상호작용) 패턴은 1D window로 안 보인다. focus는 "관심 구간만 떼어 다시 보기"(시계열 기간 brush와 **동일 의미로 통일** — 사용자 혼란 방지).
- **wafer 툴팁(요청)**: 점 hover 시 `wafer_id·root_lot_id·tkout`(+ value). yFeat==valueField면 값 중복 표기 제거.
- **가정·한계**: 점 ≤5000 다운샘플(자원). focus는 사각 영역만.

### 7.2 Heatmap (2D binned aggregate)
- **무엇**: x_bin×y_bin 격자, cell 색=값의 평균/중앙값. 저표본 셀 디엠퍼시스.
- **도입 배경**: scatter는 밀집 시 패턴이 뭉갬 → 격자 집계로 "어느 (x,y) 조합이 좋은/나쁜 구간인지" 요약.
- **가정·한계(중요)**: bin 수에 민감. 저표본 셀은 우연 변동(그래서 흐리게). **heatmap·rank는 서버가 내려준 scatter 점(7.1)으로 프론트에서 계산** → 점 수가 `SCATTER_CAP`(5000) 이하면 전수, 초과면 **샘플 기반 집계**(전수 아님)다. 실데이터가 5000점을 넘으면 "어느 구간이 best/worst"인 rank가 **샘플 운에 흔들릴 수 있음**(현재 데모 ≈1600점<5000이라 전수). 실데이터 적용 시 heatmap·rank를 백엔드 전수 집계로 분리 권장.

### 7.3 순위표 ↔ heatmap 연동 (요청 기능)
- **무엇**: 순위 행 hover → 대응 heatmap 셀 강조(테두리+그림자), 셀 hover → 행 강조 + 보이게 스크롤. `(x_bin,y_bin)`로 매칭.
- **도입 배경**: "순위 1위가 **지도상 어디**인가"가 표·맵 분리 시 즉시 안 보임. 연동으로 위치-순위를 즉각 대응.
- **교차검증 포인트**: 순위표 집계값이 heatmap 셀 색/값과 일치하는가?

---

## 8. 공통 상호작용 메커니즘

- **linked brushing(기간 brush → 재집계)**: 시계열에서 기간을 드래그하면 window·요약표가 **그 구간 wafer로 재집계**(시계열 자체는 전체 유지). stale 요청 폐기(시퀀스 가드).
  - 배경: "이 기간만 보면 window가 어떻게 달라지나"를 라운드트립 없이.
- **scatter wafer 식별 툴팁(요청, 공통)**: 모든 scatter의 점=wafer 1매라는 사실을 hover로 확인 가능하게. 백엔드가 점마다 식별자 동행.
- **다운샘플**: 점이 많으면 scatter는 표시용 표본(≤5000). **window/binned 집계는 전수**(백엔드, 관측-only). 단 **interaction의 heatmap·rank는 표시 점으로 프론트 계산**이라 점이 5000을 넘으면 샘플 기반이 됨(7.2 참조) — 이 경우만 "탐색용"으로 해석.

## 9. 핵심 통계 공식 (검증용 요약)

| 항목 | 공식 | 비고 |
|---|---|---|
| 단기 σ(within) | `MR̄ / 1.128`, `MR̄=mean(|xᵢ−xᵢ₋₁|)` | I-MR, d2(n=2)=1.128 |
| 전체 σ(overall) | 표본 표준편차 | ddof=1 |
| 관리한계 | `μ ± 3σ` | σ=관측 target 표본 std |
| Cpk | `min(USL−μ, μ−LSL)/(3·σ_within)` | 군내 능력 |
| Ppk | `min(USL−μ, μ−LSL)/(3·σ_overall)` | 장기 능력 |
| in-spec% | `Φ((U−μ)/σ) − Φ((L−μ)/σ)` | 정규근사 |
| 추정 | OLS `ŷ=a·x+b`(관측 적합) → 미관측 x로 예측 | R² 동반 |
| drift shift | `(mean(최근 k) − mean(앞 n−k))/σ`, `k=max(5,⌊n/5⌋)` | n≥20, flag |Δ|≥1 |
| 관리이탈 예보 | `#{ŷ ∉ [LCL,UCL]}`, `shift=(mean ŷ − mean y)/σ` | 한계=관측 target 기준. 신뢰 가드: `R²<0.2 ∨ 외삽≥0.5` → 참고용 강등 |
| 외삽 비율 | `mean(추정 X ∉ [관측 X min, max])` | 회귀 외삽 정도(신뢰도 보조) |
| 상관(driver) | Pearson `r(feature,target)` | n≥5, |r| 정렬 |

## 10. 교차검증 체크리스트 (다른 모델용)

다른 모델은 아래 질문에 "타당/의문/반증"으로 답하며 설계를 검증할 수 있다.

1. **lag 모델**: target을 fab 후 50~60일에 확보한다는 전제가 모든 "추정/예보/gap"의 근간이다. 이 전제가 틀리면 어떤 결론이 무너지는가?
2. **관측-only 편향**: 최근 구간은 먼저 확보된 wafer만 보임(생존 편향). window·관리한계가 이 편향을 받는가? 추정이 이를 충분히 보완하는가?
3. **관리한계 vs spec**: 별도 target spec 없이 "관측 target ±3σ"를 이탈 기준으로 쓰는 게 타당한가? 어떤 경우 위험한가?
4. **추정 외삽**: 최근 feature가 관측 X 범위를 벗어났을 때(drift) 동시점 회귀 외삽의 신뢰도는? R² 외 어떤 지표를 봐야 하나?
5. **drift 분할(80/20)**: baseline 80% / recent 20%, |Δ|≥1σ flag — 둔감/과민 균형이 적절한가? 계절성·점진 추세를 놓치나?
6. **Cpk에 단기σ**: within-σ(MR)가 이 데이터(시간 정렬·혼합 lot)에서 적절한가? subgroup 정의가 타당한가?
7. **상관→의사결정**: driver 랭킹은 상관(비인과·선형)인데, 이를 "원인"처럼 쓰는 UI 표현이 과한가?
8. **조건 비교**: 관측 데이터 평균 줄세움이 교란요인(설비-제품 혼합)을 무시하는가? 언제 오도하나?
9. **정규근사**: in-spec%·관리한계의 정규성 가정이 깨지는 분포에서 표시가 오도할 수 있는가?
10. **데모 데이터 주의**: 현재 수치는 driver feature에 인위적 상승추세를 주입한 **시나리오 재현용 합성 데이터**다(실측 아님). 결론의 "크기"가 아니라 "메커니즘"을 검증하라.
